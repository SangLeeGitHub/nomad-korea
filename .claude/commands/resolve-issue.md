# GitHub 이슈 해결 계획

당신은 GitHub 이슈를 분석하고 체계적인 해결 계획을 수립하는 전문 소프트웨어 아키텍트입니다.

## 대상 이슈 번호

$ARGUMENTS

---

## 실행 프로세스

### 1단계: 이슈 가져오기

`gh issue view $ARGUMENTS` 명령어를 사용하여 GitHub에서 이슈 정보를 가져옵니다.

### 2단계: 이슈 분석

가져온 이슈를 다음 관점에서 분석하세요:

1. **이슈 유형 파악**
   - bug: 버그 수정
   - feature: 새로운 기능 추가
   - enhancement: 기존 기능 개선
   - refactor: 코드 리팩토링
   - documentation: 문서 작업

2. **핵심 요구사항 추출**
   - 이슈에서 요구하는 핵심 기능/수정 사항
   - 성공 조건 (Definition of Done)
   - 제약 사항 또는 주의사항

3. **영향 범위 파악**
   - 관련 파일 및 컴포넌트 식별
   - 의존성 분석
   - 사이드 이펙트 가능성

### 3단계: 코드베이스 탐색

이슈와 관련된 코드를 탐색하여:
- 수정이 필요한 파일 목록 작성
- 기존 코드 패턴 파악
- 재사용 가능한 유틸리티/컴포넌트 확인

### 4단계: 상세 해결 계획 수립

다음 형식으로 구체적인 해결 계획을 작성하세요:

```markdown
## 이슈 요약
- **이슈 번호**: #[번호]
- **제목**: [이슈 제목]
- **유형**: [bug/feature/enhancement/refactor/documentation]
- **우선순위**: [high/medium/low]

## 문제 분석
[이슈의 핵심 문제점 또는 요구사항 설명]

## 해결 전략
[선택한 접근 방식과 그 이유]

## 구현 계획

### 작업 1: [작업 제목]
- **파일**: `path/to/file.ts`
- **작업 내용**:
  - [구체적인 변경 사항 1]
  - [구체적인 변경 사항 2]
- **예상 영향**: [이 변경으로 인한 영향]

### 작업 2: [작업 제목]
- **파일**: `path/to/another-file.ts`
- **작업 내용**:
  - [구체적인 변경 사항]
- **예상 영향**: [이 변경으로 인한 영향]

[필요한 만큼 작업 항목 추가]

## 테스트 계획
- [ ] [테스트 항목 1]
- [ ] [테스트 항목 2]
- [ ] [테스트 항목 3]

## 위험 요소 및 대응 방안
| 위험 요소 | 가능성 | 영향도 | 대응 방안 |
|---------|--------|-------|---------|
| [위험 1] | 낮음/중간/높음 | 낮음/중간/높음 | [대응책] |

## 예상 수정 파일 목록
- `path/to/file1.ts` - [변경 사유]
- `path/to/file2.ts` - [변경 사유]
```

### 5단계: 사용자 검토 요청

계획 작성이 완료되면 사용자에게 검토를 요청하세요:

"위 계획을 검토해 주세요. 수정이 필요한 부분이나 추가 고려사항이 있으면 알려주세요. 승인하시면 구현을 시작하겠습니다."

---

## 주의사항

- 모든 계획은 한국어로 작성
- 파일 경로는 정확하게 명시
- 각 작업은 독립적으로 검증 가능하도록 분리
- 기존 코드 스타일과 패턴을 준수
- 사용자 승인 전까지 실제 코드 수정 금지
